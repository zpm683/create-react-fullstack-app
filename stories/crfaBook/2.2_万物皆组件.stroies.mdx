## 万物皆组件

其实在前端的世界里，提出“组件”的概念之前已经有一个被称为“Atomic Design 原子化设计模式”的概念了。  
 这是在 2013 年由网页设计师 Brad Frost 从化学(元素周期表)中受到启发所提出的。

- 基本理念？
  >     原子（Atoms）结合在一起，形成分子（Molecures），进一步结合形成的生物体（Organisms），
  >     各种生物体聚合为模板（Templates），模板填充数据后形成页面（Pages）。
- 如何理解？
  > 原子（Atoms）: 在 Web 前端的世界里，UI 上最小的东西就是单个 HTML 元素了，我们便把单个 HTML 元素（div/input/button 等）看作一个原子，  
  >  分子（Molecures）: 多个原子组合而成的东西称为分子,比如一个 input 输入框和一个 button 按钮（类似百度主页的搜索框与检索按钮）  
  >  生物体（Organisms）: 分子与原子，分子与分子的组合物，比如再刚才的输入框与按钮分子的基础上再加上一个 LOGO 图片（img 原子）  
  >  模板（Templates）: 多个生物体组合而成，比如组成 UI 顶部的应用栏，UI 中部的核心功能区，UI 底部的导航等。但是要注意的是此时并没有具体数据，比如各个原子上的文字，图片等  
  >  页面（Pages）: 在模板的基础上添加每个原子所需要的数据（文字/图片等等）。
- 与组件化的关系？
  > 不难看出，React 提出的万物皆组件其实就是原子化设计模式的概括抽象版。  
  >  React 的世界观中，无论是“原子”还是“页面”均称为组件，这些组件的区别就是组件的规模了。  
  >  我们往往称呼“页面”为一个“顶层组”件，“模板”为“根组件”，而更小的内容则称之为“子组件”。  
  >  React 为何不直接使用原子化设计的概念？我觉得这是因为自从 React 开源后，就希望依托广大的开源社区提供各种“可插拔”式支持来打造一个强大的 React 生态圈，  
  >  对于一般的开发者而言我们希望使用“灵活而简洁”的开源模块，我们不希望使用一个库时，去学习太多的复杂的概念，这会带来一些心智负担。  
  >  好比我们使用一些 React 社区开源的组件库时，我们关注的是每个组件提供的功能，这时候每个组件其实就是一个“原子”。  
  >  但实际上，它们的底层可能是更多的原子（HTML 元素）组成，可见开发这些组件库的成员和使用这些库的人员的角度是不同的：  
  >  从原子化角度：
  >
  > - 开发这些组件库的成员：我写的是一个生物体，由小的原子（HTML 元素）组成。
  > - 使用这些组件库的成员：我用的是一个原子，我要用它来打造自己的组织。  
  >   从组件化角度：
  > - 开发这些组件库的成员：我写的是一个公共组件，由小的内部组件组成。
  > - 使用这些组件库的成员：我用的是一个公共组件，我要用它来打造自己的根组件。  
  >   所以，你可以发现，当你把这些统称为组件的时候，可以大大降低心智负担。
